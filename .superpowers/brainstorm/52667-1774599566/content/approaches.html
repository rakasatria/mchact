mchact-memory: Architecture Approaches

How should the Honcho-inspired observation engine integrate with microclaw's existing memory + FTS5 + MoA stack?

A

Layered Crate — PostgreSQL Observation Engine

New crates/mchact-memory/ crate backed by PostgreSQL+pgvector. Sits alongside existing SQLite storage. The observation engine (deriver/dreamer) writes to PostgreSQL. Existing SQLite flat memories continue for backward compat. MemoryProvider trait implementation queries both stores and merges results.

Pros

  • Zero risk to existing features (FTS5, compressor, MoA all untouched)
  • PostgreSQL+pgvector for proper vector search from day one
  • Can migrate incrementally — old flat memories still work
  • Clean separation: SQLite for sessions/messages, PostgreSQL for observations

Cons

  • Two databases to manage (SQLite + PostgreSQL)
  • Query merging adds complexity
  • Dual-write during migration period
B

Full PostgreSQL Migration — Replace SQLite Entirely

Move everything to PostgreSQL: messages, sessions, auth, memories, observations. The existing microclaw-storage crate gets a PostgreSQL backend (sqlx). FTS5 replaced by PostgreSQL full-text search + pgvector. Observations are the new memory model.

Pros

  • Single database — no split-brain
  • PostgreSQL FTS + pgvector hybrid search (like Honcho's RRF)
  • Cleaner long-term architecture
  • Transactions span messages + observations atomically

Cons

  • Massive migration — touches every DB operation
  • Breaks zero-config SQLite deployment (now requires PostgreSQL)
  • FTS5 work just completed — would be replaced
  • High risk, long timeline
C

Dual-Backend with PostgreSQL Primary Path

New crates/mchact-memory/ uses PostgreSQL+pgvector for the observation engine. Existing SQLite stays for sessions/messages/FTS5. But add a config toggle: storage_backend: "sqlite" | "postgres". When postgres, ALL storage migrates. When sqlite, only observations use PostgreSQL. Future: deprecate sqlite path.

Pros

  • Observations get PostgreSQL+pgvector immediately
  • SQLite still works for lightweight/local deployments
  • Clear migration path: sqlite → postgres toggle
  • Preserves existing FTS5/compressor/MoA work
  • Recommended approach

Cons

  • Two code paths to maintain initially
  • Config complexity
