Flow 2: Agent Searches Memory

SQLite Path
Tool call: structured_memory_search
  query: "what language does user prefer"
          │
          ▼
ObservationStore::search(query, strategy)
          │
          ▼
SqliteStore::search()
  1. capabilities().hybrid_search?
     → false (unless sqlite-vec)
  2. FTS5 keyword search:
     SELECT * FROM memories
     WHERE content LIKE '%rust%'
     OR content LIKE '%language%'
     ORDER BY updated_at DESC
     LIMIT 50
  3. Optional: sqlite-vec KNN
     if feature enabled + embedding exists
     → re-rank by cosine distance
  4. Return results as Vec<Observation>
     (all level=Explicit, no peer info)

Result:
  [id=5] [PROFILE] user is backend dev,
         prefers Rust
PostgreSQL Path
Tool call: structured_memory_search
  query: "what language does user prefer"
          │
          ▼
ObservationStore::search(query, strategy)
          │
          ▼
PgStore::search() → RRF Hybrid
  1. Semantic arm:
     embed(query) → Vec<f32>
     SELECT id, embedding <=> $1 AS dist
     FROM observations
     WHERE is_archived = false
     ORDER BY dist LIMIT 20
     → ranked by vector similarity

  2. Keyword arm:
     SELECT id, ts_rank(tsv, query) AS r
     FROM observations
     WHERE tsv @@ plainto_tsquery($1)
     ORDER BY r DESC LIMIT 20
     → ranked by text relevance

  3. RRF merge:
     score = 1/(k+rank_semantic)
           + 1/(k+rank_keyword)
     sort by combined score
     → best of both worlds

  4. Return with full metadata:
     level, peer info, source_ids

Result:
  [id=5] [EXPLICIT] [bot→alice]
    user is backend dev, prefers Rust
  [id=8] [DEDUCTIVE] [bot→alice]
    likely familiar with tokio/async
    (derived from: #5)

Flow 4: Dreamer — Offline Consolidation (PostgreSQL Only)

Dreamer runs on schedule (e.g., every 2 hours)
scheduler.rs → dreamer loop
  if !store.capabilities().dreamer { return; }  ← SQLite skips entirely
          │
          ▼
  ObservationStore::dream(peer_id: alice)
          │
          ▼
  Phase 1: DEDUCTION SPECIALIST
    Fetch recent explicit observations for alice
    LLM prompt: "Given these facts, what can you logically infer?"
    Input:  [explicit] "allergic to peanuts"
            [explicit] "loves Thai food"
            [explicit] "eats out frequently"
    Output: [deductive] "needs to verify peanut content at Thai restaurants"
            source_ids: [obs_peanut, obs_thai]  ← DAG links
          │
          ▼
  Phase 2: INDUCTION SPECIALIST
    Fetch all observations for alice (explicit + deductive)
    LLM prompt: "What patterns do you see across these observations?"
    Input:  20+ observations over 2 weeks
    Output: [inductive] "alice is health-conscious and detail-oriented about food"
            confidence: medium
            pattern_type: personality
            source_ids: [obs_1, obs_5, obs_8, obs_12]
          │
          ▼
  Phase 3: CONTRADICTION DETECTION
    Compare new observations against existing
    LLM prompt: "Do any of these contradict each other?"
    Found:  [explicit] "prefers working alone" (March 15)
         vs [explicit] "loves pair programming" (March 22)
    Output: [contradiction] "conflicting statements about collaboration preference"
            source_ids: [obs_alone, obs_pair]
          │
          ▼
  Phase 4: CONSOLIDATION
    Find near-duplicate observations (pgvector cosine < 0.1)
    Merge: keep most detailed version, archive others
    Increment times_derived on consolidated observations
          │
          ▼
  Phase 5: PEER CARD UPDATE
    Extract top stable facts → peer_card (max 40)
    "alice: backend dev, Rust, allergic to peanuts,
     health-conscious, Thai food lover, ..."

Critical: Where Does Each Data Type Live?

Data Ownership — Both Backends
Data SQLite Backend PostgreSQL Backend
Chat messages SQLite messages SQLite messages (stays in SQLite)
Message FTS index SQLite FTS5 SQLite FTS5 (stays)
Sessions SQLite sessions SQLite sessions (stays)
Auth / API keys SQLite SQLite (stays)
Scheduled tasks SQLite SQLite (stays)
Memories / Observations SQLite memories PostgreSQL observations
Memory vectors sqlite-vec (optional) pgvector HNSW
Peers N/A PostgreSQL peers
Peer cards N/A PostgreSQL peers.peer_card
Observation queue N/A (reflector loop) PostgreSQL observation_queue
MoA findings SQLite subagent_findings SQLite subagent_findings (stays)
Dreamer runs log N/A PostgreSQL dreamer_runs

Key insight: Only memory/observations move to PostgreSQL. Everything else stays in SQLite. This means messages, sessions, auth, FTS5, MoA findings — all untouched. The mchact-memory crate only owns the green rows.

Honest Risks & Edge Cases

Where This Could Break

Risk 1: Cross-database joins

Messages are in SQLite, observations in PostgreSQL. The deriver needs to read messages (SQLite) and write observations (PostgreSQL). No cross-DB transactions.
Mitigation: Deriver reads messages via existing DB methods, then writes to PG separately. Idempotent writes — if it crashes between read and write, the queue re-processes.

Risk 2: SQLite backend silently drops data

When running on SQLite, store_observation() drops peer_ids, source_ids, level. If someone later switches to PostgreSQL, those observations lack the rich metadata.
Mitigation: Migration tool that re-processes SQLite memories through the deriver to enrich them when upgrading to PostgreSQL.

Risk 3: Two dedup paths

SQLite uses Jaccard similarity + topic heuristic. PostgreSQL uses pgvector cosine distance. Different dedup strategies may produce different results.
Mitigation: Accept this — they're different capability tiers. Document that PostgreSQL dedup is more accurate.

Risk 4: Peer auto-creation from chat_id

On PostgreSQL, we need to map chat_id + sender_name → peer. First message from a new user auto-creates a peer. If sender_name changes (Telegram display names), we might create duplicates.
Mitigation: Map by (channel, external_user_id) not display name. Telegram has stable user IDs.

Risk 5: PostgreSQL not available

If configured for PostgreSQL but DB is unreachable at startup?
Mitigation: Fall back to SQLite backend with warning. Same pattern as existing MCP fallback in memory_backend.rs.
